# PRD: Resource Registry for Reliable Cleanup

## Status

Draft backlog item.

## Problem

Sherpa destroy and clean paths rely heavily on runtime resource names containing `lab_id`. This is useful as a fallback, but it is not a robust primary source of truth. If a resource is created with a malformed name, renamed, truncated by a backend, or missed by a prefix filter, destroy/clean may leak resources. Conversely, broad substring matching can accidentally target unrelated resources if names collide.

Sherpa needs exact resource inventory and ownership tracking so cleanup can be deterministic and auditable.

## Findings

- `services/destroy.rs` matches Docker containers by checking whether container names contain `lab_id`.
- VM/domain cleanup checks whether libvirt domain names contain `lab_id`.
- Disk cleanup filters storage-pool volumes by `contains(lab_id)`.
- Docker network cleanup checks whether network names contain `lab_id`.
- Libvirt network cleanup checks whether network names contain `lab_id`.
- Host interface cleanup calls `network::find_interfaces_fuzzy(lab_id)` and then applies prefix allowlists such as bridge, veth, tap, and container-veth prefixes.
- `services/up.rs`, `services/redeploy.rs`, and `services/node_ops.rs` construct many runtime names inline with `format!(...)`.
- Some resource records exist in the database for labs, nodes, and links, but there is no unified resource inventory table that records every created runtime resource with provider/type/name/state.
- Naming conventions are spread across services and constants, not centralized as a cleanup contract.

## Goals

- Track exact resources created for each lab.
- Make destroy/clean use exact inventory first and fuzzy name matching only as a fallback.
- Reduce leaked resources after partial creation failures.
- Improve operator diagnostics for failed cleanup.
- Centralize resource naming rules and validate them.

## Non-goals

- Removing human-readable names.
- Removing all fallback cleanup by name matching.
- Replacing Docker/libvirt APIs.
- Building a general CMDB outside Sherpa's lab resources.

## Proposed solution

Add a Sherpa resource registry persisted in SurrealDB. Creation paths record each resource as it is planned/created, and cleanup paths use the registry as the primary source of truth.

Recommended resource record fields:

| Field | Purpose |
|---|---|
| `id` | Database record ID. |
| `lab_id` | Owning Sherpa lab ID. |
| `owner_username` | Owning user for audit and future policy. |
| `provider` | Docker, libvirt, network, filesystem, db, template. |
| `resource_type` | Container, Docker network, domain, disk, libvirt network, bridge, veth, tap, directory, file, etc. |
| `name` | Exact backend resource name. |
| `external_id` | Optional backend ID if available. |
| `created_phase` | Lifecycle phase that created it. |
| `state` | Planned, creating, created, delete_pending, deleted, delete_failed. |
| `metadata` | Small structured details needed for cleanup. |
| `created_at`, `updated_at` | Audit timestamps. |
| `last_error` | Last cleanup/create error if any. |

## Functional requirements

- Lab creation must register resources before or immediately after creation in a way that supports partial-failure cleanup.
- Destroy must query the resource registry and delete exact resources by provider/type/name.
- Clean must use the same registry-driven cleanup path when records exist.
- Fuzzy `lab_id` matching must remain available as a fallback for old labs and partially registered resources.
- Resource names must be generated by centralized helpers where practical.
- Cleanup summaries must report registry-driven cleanup and fallback cleanup separately.
- Registry records must be updated when cleanup succeeds or fails.

## Non-functional requirements

- Registry writes must not make resource creation significantly slower.
- Registry updates must be robust to partial failures.
- Cleanup must remain safe: exact matching should be preferred, broad matching should be conservative.
- Resource records must not store credentials or sensitive generated file contents.
- Naming helpers must account for backend length limits, especially Linux interface names.

## Action plan

### Phase 1: Inventory current resource types

- Enumerate all resources created by `up.rs`, `redeploy.rs`, and `node_ops.rs`.
- Map each resource to provider, type, name pattern, creation phase, and deletion function.
- Identify backend IDs/labels/metadata that can make cleanup more reliable:
  - Docker labels on containers/networks.
  - libvirt metadata where available.
  - DB records for links/nodes.
  - filesystem paths.

### Phase 2: Define registry schema

- Add DB schema and CRUD helpers for resource records.
- Add typed enums for provider, resource type, and lifecycle state.
- Add helper APIs for register planned, mark created, mark delete pending, mark deleted, mark failed.

### Phase 3: Centralize naming

- Move resource naming into helpers close to the resource model.
- Add tests for every naming helper, including length limits and `lab_id` inclusion.
- Replace scattered inline `format!` usage incrementally.

### Phase 4: Creation integration

- In `up_lab`, register resources by phase before creation where rollback needs to know them.
- Mark resources created after successful backend calls.
- On resource-phase failure, use registry records for cleanup.

### Phase 5: Destroy/clean integration

- Update `destroy_lab` to use registry records first.
- Keep existing fuzzy cleanup as fallback and mark fallback-discovered resources in the summary.
- Update `clean_lab` to share the same cleanup engine where possible.
- Update cleanup summaries and errors to expose exact provider/type/name.

### Phase 6: Migration/backward compatibility

- Existing labs will not have registry records.
- For old labs, destroy should fall back to current fuzzy name matching.
- Optionally add an admin scan/import action that discovers resources for existing labs and backfills registry entries.

## Testing plan

- Unit tests for naming helpers.
- DB tests for registry state transitions.
- Unit tests for cleanup ordering and fallback behavior using fake resource inventories.
- Integration tests for partial failure cleanup where resource records exist.
- Regression tests proving unrelated resources with similar names are not deleted when exact registry data is available.
- Backward-compatibility test for old labs without registry records using fallback matching.

## Acceptance criteria

- New lab creation records runtime resources in SurrealDB.
- Destroy uses exact registry records before fuzzy matching.
- Clean shares registry-driven cleanup behavior or uses the same cleanup engine.
- Fallback cleanup still works for old labs.
- Cleanup summaries identify exact resources cleaned and failed.
- Naming helper tests cover all resource name formats used by create/redeploy/destroy.

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| Registry gets out of sync with runtime | Scanner/admin scan can reconcile; destroy uses fallback discovery. |
| Register-before-create leaves planned records for failed resources | Track lifecycle state and cleanup only created/delete_pending records unless planned resource needs explicit rollback. |
| Backend name limits conflict with central naming | Add tests and explicit truncation/hash strategy. |
| Cleanup becomes too complex | Implement one provider at a time and keep fallback behavior during migration. |
